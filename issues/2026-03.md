## [MCP server that reduces Claude Code context consumption by 98%](https://mksg.lu/blog/context-mode)

# Claude Code 컨텍스트 사용량을 98% 줄이는 MCP 서버

이 글은 Claude Code와 같은 AI 코딩 어시스턴트의 가장 큰 문제점인 컨텍스트 토큰 소비를 획기적으로 줄이는 MCP(Model Context Protocol) 서버 구현을 소개합니다. 기존 방식은 전체 파일 내용을 컨텍스트에 포함시켜 토큰을 빠르게 소진하는 반면, 이 솔루션은 "컨텍스트 모드"를 도입하여 파일의 구조적 정보(함수 시그니처, 클래스 정의 등)만 먼저 제공하고, 필요한 경우에만 전체 내용을 요청하도록 설계되었습니다. 이를 통해 98%의 컨텍스트 절감 효과를 달성했습니다.

AI Agent 개발자 관점에서 이 접근법은 매우 중요한 시사점을 제공합니다. 대규모 코드베이스를 다루는 Agent는 항상 컨텍스트 윈도우 제한과 비용 문제에 직면하는데, 이 MCP 서버는 "lazy loading" 전략을 통해 이 문제를 해결합니다. 특히 MCP 프로토콜을 활용하여 Agent가 필요에 따라 점진적으로 정보를 요청할 수 있는 구조를 만들었다는 점이 핵심입니다. 이는 단순히 Claude Code뿐만 아니라, 코드 분석, 리팩토링, 문서 생성 등 다양한 AI Agent 시나리오에 적용 가능한 패턴으로, 컨텍스트 효율성이 중요한 모든 Agent 개발에 참고할 만한 실용적 사례입니다.

AI 에이전트 개발자들에게 이 자료가 중요한 이유는 컨텍스트 관리가 에이전트의 성능과 신뢰성을 결정하는 핵심 요소이기 때문입니다. 에이전트가 대화 히스토리, 작업 상태, 외부 도구 호출 결과 등을 효과적으로 추적하고 활용하지 못하면 일관성 없는 응답이나 작업 실패로 이어질 수 있습니다. 이 컬렉션은 이러한 문제를 해결하기 위한 구체적인 패턴과 기법들을 제시합니다.

특히 프로덕션 환경에서 에이전트 시스템을 최적화하거나 디버깅할 때 실질적인 도움을 받을 수 있습니다. 멀티 에이전트 간 컨텍스트 공유, 메모리 관리 전략, 컨텍스트 윈도우 최적화 등 실무에서 자주 마주치는 기술적 과제들에 대한 솔루션을 제공하여, 개발자들이 더 안정적이고 효율적인 에이전트 시스템을 구축할 수 있도록 지원합니다.

## [“Car Wash” test with 53 models](https://opper.ai/blog/car-wash-test)

# "Car Wash" 테스트로 53개 AI 모델 평가

이 연구는 AI 모델들이 실제 세계의 모호한 지시사항을 얼마나 잘 이해하고 처리하는지 평가하기 위해 "Car Wash" 테스트를 개발했습니다. 이 테스트는 "세차장에 가는 길에 우유를 사오세요"와 같은 일상적이지만 암묵적인 논리를 요구하는 지시사항을 53개의 다양한 AI 모델에 제시했습니다. 핵심은 모델이 세차장에 먼저 가야 하는지, 우유를 먼저 사야 하는지 올바른 순서를 추론할 수 있는지 평가하는 것입니다. 이는 벤치마크 점수로는 측정하기 어려운 상식적 추론 능력을 테스트합니다.

테스트 결과, 최신 대형 모델들(GPT-4, Claude 3.5 Sonnet 등)은 비교적 높은 정확도를 보였지만, 많은 모델들이 여전히 이러한 기본적인 논리적 순서 추론에서 실패했습니다. AI Agent 개발자들에게 이는 매우 중요한 시사점을 제공합니다. 실제 프로덕션 환경에서 에이전트가 다단계 작업을 수행할 때, 단순히 벤치마크 성능이 좋은 모델을 선택하는 것만으로는 충분하지 않습니다. 특히 사용자의 암묵적 의도를 파악하고 논리적 순서를 결정해야 하는 작업 계획(task planning) 단계에서는 이러한 상식 추론 능력이 필수적입니다.

개발자들은 자신의 특정 유스케이스에 맞는 유사한 실용적 테스트를 설계하여 모델을 선택해야 합니다. 이 연구는 모델 선택 시 단순 성능 지표를 넘어 실제 시나리오 기반 평가의 중요성을 강조하며, 특히 멀티스텝 에이전트 워크플로우를 구축할 때 각 모델의 추론 한계를 명확히 이해하고 적절한 프롬프트 엔지니어링이나 검증 로직을 추가해야 함을 보여줍니다.

## [Mercury 2: Fast reasoning LLM powered by diffusion](https://www.inceptionlabs.ai/blog/introducing-mercury-2)

# Mercury 2: 확산 모델 기반 고속 추론 LLM

Inception Labs가 공개한 Mercury 2는 기존의 자기회귀(autoregressive) 방식 대신 확산(diffusion) 모델을 활용한 새로운 추론 LLM입니다. 전통적인 LLM이 토큰을 순차적으로 하나씩 생성하는 반면, Mercury 2는 확산 과정을 통해 전체 응답을 병렬적으로 생성함으로써 추론 속도를 크게 향상시켰습니다. 이는 특히 긴 응답이나 복잡한 추론이 필요한 작업에서 레이턴시를 획기적으로 줄일 수 있는 가능성을 제시합니다.

AI Agent 개발자들에게 이 기술은 실시간 상호작용이 중요한 애플리케이션에서 큰 가치를 제공합니다. 기존 추론 모델의 경우 체인-오브-씽킹(chain-of-thought) 과정에서 긴 대기 시간이 발생했지만, Mercury 2의 병렬 생성 방식은 에이전트의 의사결정 속도를 높이고 사용자 경험을 개선할 수 있습니다. 특히 멀티 에이전트 시스템이나 실시간 도구 호출이 빈번한 워크플로우에서 전체 파이프라인의 처리량을 증가시킬 수 있습니다.

다만 확산 기반 언어 모델은 아직 초기 단계 기술로, 품질과 제어 가능성 측면에서 검증이 필요합니다. 개발자들은 Mercury 2가 자신의 에이전트 유스케이스에서 기존 모델 대비 어떤 트레이드오프를 가져오는지 벤치마킹해볼 필요가 있으며, 특히 추론 정확도와 속도 간의 균형을 평가해야 합니다.

## [abhigyanpatwari/GitNexus](https://github.com/abhigyanpatwari/GitNexus)

# GitNexus: 브라우저 기반 코드 지식 그래프 생성 도구

GitNexus는 서버 없이 완전히 클라이언트 측에서 동작하는 코드 인텔리전스 엔진입니다. GitHub 저장소 URL이나 ZIP 파일을 입력하면 브라우저 내에서 즉시 대화형 지식 그래프를 생성하며, 내장된 Graph RAG Agent를 통해 코드베이스를 탐색할 수 있습니다. 서버 인프라가 필요 없다는 점에서 데이터 프라이버시와 배포 복잡도 측면에서 큰 장점을 제공합니다.

AI Agent 개발자들에게 이 도구는 특히 유용합니다. 코드 분석 및 이해를 위한 RAG(Retrieval-Augmented Generation) 파이프라인을 구축할 때, GitNexus의 그래프 기반 접근 방식은 코드 간의 관계와 의존성을 구조화된 형태로 제공합니다. 이는 단순 벡터 검색보다 더 정확한 컨텍스트를 Agent에게 제공할 수 있으며, 코드 탐색, 리팩토링 제안, 버그 추적 등의 작업을 수행하는 AI Agent 개발에 직접 활용할 수 있습니다.

특히 클라이언트 측에서 완전히 동작한다는 특성은 민감한 코드베이스를 다루는 엔터프라이즈 환경에서 중요합니다. 외부 서버로 코드를 전송하지 않고도 지식 그래프를 구축하고 RAG Agent를 실행할 수 있어, 보안이 중요한 프로젝트에서도 안전하게 사용할 수 있습니다. 오픈소스로 공개되어 있어 자체 AI Agent 시스템에 통합하거나 커스터마이징하는 것도 가능합니다.

## [Show HN: Llama 3.1 70B on a single RTX 3090 via NVMe-to-GPU bypassing the CPU](https://github.com/xaskasdf/ntransformer)

# Llama 3.1 70B를 단일 RTX 3090에서 실행하는 혁신적 접근법

이 프로젝트는 NVMe SSD에서 GPU로 직접 데이터를 전송하여 CPU와 시스템 RAM을 우회하는 방식으로 70B 파라미터 규모의 Llama 3.1 모델을 단일 RTX 3090(24GB VRAM)에서 실행할 수 있게 합니다. 기존 방식은 모델 가중치를 CPU 메모리에 먼저 로드한 후 GPU로 전송해야 했지만, 이 기술은 PCIe 버스를 통해 NVMe에서 GPU VRAM으로 직접 전송함으로써 병목 현상을 크게 줄였습니다. 이를 통해 140GB 이상의 모델을 24GB VRAM만으로 구동할 수 있게 되었습니다.

AI Agent 개발자들에게 이 기술은 매우 실용적인 의미를 갖습니다. 고가의 A100이나 H100 없이도 일반 소비자용 GPU로 대규모 언어 모델을 로컬에서 실행할 수 있어, 개발 및 테스트 비용을 대폭 절감할 수 있습니다. 특히 프라이버시가 중요한 엔터프라이즈 AI Agent나 오프라인 환경에서 작동해야 하는 에이전트 개발 시 클라우드 API 의존도를 낮출 수 있습니다. 다만 추론 속도는 클라우드 솔루션보다 느릴 수 있으므로, 응답 시간이 중요하지 않은 배치 처리나 백그라운드 작업에 적합합니다.

기술적으로는 GPU Direct Storage와 유사한 개념을 활용하며, 레이어별 스트리밍 방식으로 필요한 가중치만 순차적으로 로드합니다. 이는 메모리 제약이 있는 환경에서 대규모 모델을 활용하려는 개발자들에게 새로운 아키텍처 설계 가능성을 제시하며, 특히 멀티모달 AI Agent나 복잡한 추론이 필요한 작업에서 로컬 배포 옵션을 확대합니다.

## [Netflix/metaflow](https://github.com/Netflix/metaflow)

# Netflix Metaflow - AI/ML 시스템 구축 및 배포 프레임워크

Netflix가 개발한 Metaflow는 AI/ML 시스템을 구축, 관리, 배포하기 위한 오픈소스 프레임워크입니다. 이 도구는 데이터 과학자와 ML 엔지니어가 프로덕션 환경에서 실제로 작동하는 머신러닝 워크플로우를 빠르게 개발할 수 있도록 설계되었습니다. Python 기반의 직관적인 API를 제공하며, 로컬 개발부터 클라우드 스케일 배포까지 동일한 코드로 처리할 수 있는 것이 핵심 특징입니다. 특히 실험 추적, 버전 관리, 데이터 아티팩트 관리 등이 자동으로 처리되어 개발자가 비즈니스 로직에만 집중할 수 있습니다.

AI Agent 개발자에게 Metaflow는 복잡한 에이전트 워크플로우를 체계적으로 관리할 수 있는 강력한 도구입니다. 에이전트의 의사결정 파이프라인, 모델 학습 및 업데이트 프로세스, 다양한 데이터 소스 통합 등을 DAG(Directed Acyclic Graph) 형태로 정의하고 실행할 수 있습니다. AWS, Azure, Kubernetes 등 다양한 인프라와의 통합을 지원하며, 각 실행 단계의 결과물과 메타데이터가 자동으로 저장되어 디버깅과 재현성이 보장됩니다. 또한 @step 데코레이터를 통한 간단한 병렬 처리와 리소스 할당 기능은 대규모 에이전트 시스템의 효율적인 운영을 가능하게 합니다.

Netflix의 프로덕션 환경에서 검증된 이 프레임워크는 수천 개의 ML 워크플로우를 안정적으로 운영한 실적을 바탕으로, AI Agent의 개발-테스트-배포 사이클을 크게 단축시킬 수 있습니다. 특히 에이전트의 지속적인 학습과 개선이 필요한 환경에서, Metaflow의 자동화된 실험 관리와 버전 컨트롤 기능은 개발 생산성을 극대화하는 핵심 도구가 될 수 있습니다.

## [How I use Claude Code: Separation of planning and execution](https://boristane.com/blog/how-i-use-claude-code/)

# Claude Code 활용법: 계획과 실행의 분리

이 글은 AI 코딩 도구인 Claude Code를 효과적으로 사용하기 위한 핵심 전략으로 "계획(planning)"과 "실행(execution)"의 분리를 제안합니다. 저자는 Claude에게 즉시 코드를 작성하도록 요청하는 대신, 먼저 구현 계획을 수립하도록 하고 이를 검토한 후 실행 단계로 넘어가는 2단계 워크플로우를 사용합니다. 이 방식은 AI가 잘못된 방향으로 코드를 작성하는 것을 방지하고, 개발자가 전체 아키텍처와 접근 방식을 사전에 검증할 수 있게 합니다. 특히 복잡한 기능 구현이나 리팩토링 작업에서 이 방법론은 시행착오를 크게 줄여줍니다.

AI Agent 개발자들에게 이 접근법은 매우 실용적인 가치를 제공합니다. Agent 시스템에서도 동일한 원칙을 적용할 수 있는데, Agent가 작업을 수행하기 전에 먼저 실행 계획을 생성하고 이를 검증하는 단계를 추가하면 오류율을 낮추고 예측 가능성을 높일 수 있습니다. 저자는 계획 단계에서 파일 구조, 수정할 함수, 예상되는 부작용 등을 명확히 하도록 프롬프트를 설계하며, 이는 ReAct 패턴이나 Chain-of-Thought와 유사한 구조화된 추론 방식입니다. 이러한 메타인지적 접근은 AI Agent의 신뢰성과 디버깅 가능성을 향상시키는 핵심 패턴으로, 프로덕션 환경에서 Agent를 운영할 때 필수적인 설계 원칙이 될 수 있습니다.

# Wolfram 기술을 LLM 시스템의 기반 도구로 활용하기

Stephen Wolfram은 Wolfram 기술 스택(Wolfram Language, Mathematica, Wolfram|Alpha 등)을 LLM 시스템의 기반 도구로 통합하는 방안을 제시합니다. LLM은 자연어 처리와 패턴 인식에 뛰어나지만, 정확한 계산, 구조화된 지식 검색, 복잡한 알고리즘 실행에는 한계가 있습니다. Wolfram 기술은 이러한 약점을 보완할 수 있는 강력한 계산 엔진과 지식 베이스를 제공하며, LLM이 단순히 텍스트를 생성하는 것을 넘어 실제로 신뢰할 수 있는 계산과 분석을 수행할 수 있게 합니다.

AI Agent 개발자들에게 이는 매우 실용적인 접근법입니다. LLM을 function calling이나 tool use 패턴과 결합할 때, Wolfram API를 외부 도구로 통합하면 수학적 계산, 데이터 시각화, 과학적 분석, 실시간 데이터 조회 등의 기능을 에이전트에 추가할 수 있습니다. 예를 들어, 사용자가 복잡한 미적분 문제를 요청하거나 실시간 날씨 데이터 분석을 원할 때, LLM은 자연어를 이해하고 적절한 Wolfram 함수를 호출하여 정확한 결과를 반환할 수 있습니다.

이 접근법의 핵심 가치는 LLM의 확률적 특성과 Wolfram의 결정론적 계산 능력을 결합하여 하이브리드 AI 시스템을 구축할 수 있다는 점입니다. 특히 금융, 과학 연구, 엔지니어링 등 정확성이 중요한 도메인에서 AI Agent를 개발할 때, Wolfram 통합은 hallucination 문제를 줄이고 신뢰성을 크게 향상시킬 수 있습니다. Wolfram은 API와 플러그인 형태로 다양한 LLM 플랫폼과의 통합을 지원하고 있어, 개발자들은 비교적 쉽게 이 기능을 자신의 에이전트 시스템에 추가할 수 있습니다.

# LiteLLM: 100개 이상의 LLM API를 통합하는 AI Gateway

LiteLLM은 100개 이상의 LLM API를 OpenAI 형식으로 통일하여 호출할 수 있게 해주는 Python SDK이자 프록시 서버입니다. AI Agent 개발에서 특히 유용한 기능은 비용 추적(cost tracking), 로드밸런싱, 가드레일(guardrails) 기능입니다. 복잡한 Agent 시스템에서는 수많은 LLM 호출이 발생하는데, LiteLLM을 통해 실시간으로 API 비용을 모니터링하고, 여러 모델 간 트래픽을 자동 분산하며, 안전 장치를 설정할 수 있습니다. 또한 상세한 로깅 기능으로 Agent의 의사결정 과정을 추적하고 디버깅할 수 있어, 프로덕션 환경에서 안정적인 AI Agent 운영이 가능합니다.

HuggingFace, VLLM, NVIDIA NIM 등 오픈소스 및 자체 호스팅 모델도 지원하여, 클라우드 API와 온프레미스 모델을 혼합하여 사용하는 하이브리드 아키텍처 구축이 용이합니다. 이는 비용 최적화와 데이터 프라이버시가 중요한 엔터프라이즈 AI Agent 개발에 특히 적합합니다.
